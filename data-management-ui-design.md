## 1. 研究总结

### 1.1 联网搜索发现

#### 批量导入 UI 设计要点

基于以下参考资料的分析：
- Smart Interface Design Patterns - Bulk Import UX
- Smashing Magazine - Designing Data Importer
- CSVBox Blog - File Upload Patterns (2026)
- Medium - Bulk Upload Feature Case Study

**关键发现：**

1. **分步导入流程是行业标准**
   - 文件上传 → 数据预览 → 字段映射 → 验证 → 确认
   - 每一步都要有清晰的状态指示
   - 用户可以随时返回上一步修改

2. **拖拽上传是必备功能**
   - 虚线边框的放置区域
   - 支持点击选择文件作为备选
   - 文件类型和大小限制的清晰提示

3. **自动字段映射**
   - 自动识别常见字段名（如 name → 姓名）
   - 提供下拉选择手动映射
   - 标记必填和可选字段

4. **实时验证**
   - 逐行验证，即时显示错误
   - 错误高亮显示，提供修复建议
   - 支持部分导入（跳过错误行）

5. **进度反馈**
   - 显示导入进度条
   - 显示成功/失败数量
   - 提供详细的导入报告

#### 数据验证 UI 设计

基于参考资料：
- react-spreadsheet-import (GitHub)
- OneSchema - Advanced CSV Import Features
- UseCSV - Data Import Platform

**关键发现：**

1. **验证层级**
   - 格式验证（邮箱格式、手机号格式）
   - 业务验证（重复数据、关联数据）
   - 自定义验证（用户定义规则）

2. **错误展示**
   - 行级别错误：整行高亮
   - 单元格级别错误：单元格红色边框
   - 错误信息：悬停显示详细说明

3. **修复流程**
   - 支持原地编辑修复
   - 支持批量替换
   - 支持跳过该行

#### 企业级应用 UI 模式

基于参考资料：
- SAP Fiori Design Guidelines (List Report, Object Page)
- Apple Human Interface Guidelines (Data Entry)
- Stephanie Walter - Enterprise Data Tables
- Pencil & Paper - Data Table UX Patterns

**关键发现：**

1. **SAP Fiori Floorplans**
   - List Report: 列表页面，支持搜索、过滤、排序
   - Object Page: 详情页面，使用 Tabs 组织内容
   - Shell Bar: 顶部导航栏，包含全局操作

2. **Apple HIG 数据录入原则**
   - 让数据录入变得简单，减少错误
   - 使用智能默认值
   - 提供即时反馈

3. **数据表格设计**
   - 响应式设计，移动端使用卡片视图
   - 列宽可调整，支持水平滚动
   - 行内编辑 vs 模态编辑的选择

#### 反馈机制设计

基于参考资料：
- Mobbin - Toast UI Design
- LogRocket - Toast Notifications
- Fluent 2 Design System - Toast
- NN/G - Confirmation Dialogs

**关键发现：**

1. **Toast 通知**
   - 成功：绿色，自动消失（3秒）
   - 错误：红色，需要手动关闭
   - 进行中：显示进度

2. **确认对话框**
   - 删除操作必须确认
   - 显示操作的影响范围
   - 使用具体操作词（不用 Yes/No）

### 1.2 最佳实践总结

#### 零培训成本设计原则

| 原则 | 实现方式 |
|------|----------|
| 熟悉性 | 使用用户熟悉的 Google/Apple/SAP 交互模式 |
| 可发现性 | 操作入口明显，不需要解释 |
| 一致性 | 所有界面遵循相同的设计语言 |
| 反馈性 | 每个操作都有即时的视觉反馈 |
| 容错性 | 关键操作可撤销，错误有修复路径 |

#### 三大设计系统参考

**Google Material Design 3**
- 强调层级和动效
- 使用 FAB（浮动操作按钮）突出主要操作
- 表单使用 outlined 样式

**Apple Human Interface Guidelines**
- 强调清晰和优雅
- 使用 native 控件风格
- 表单分组明确

**SAP Fiori**
- 企业级应用标准
- Shell Bar + List Report + Object Page 结构
- 强调数据密集型界面

```
├── 分组页面 (Grouping)

    ├── 导入历史 (Import History Page)
    └── 导出数据 (Export Dialog)
```

### 2.3 操作路径

#### 场景 1: 批量导入新球员

```
首页 → Shell Bar [数据管理] → 导入新球员
     → Step 1: 选择文件 (拖拽/点击)
     → Step 2: 数据预览 (表格展示前10行)
     → Step 3: 字段映射 (自动映射 + 手动调整)
     → Step 4: 数据验证 (错误高亮 + 修复)
     → Step 5: 导入确认 (显示统计)
     → 完成 → Toast 通知 + 跳转到分组页面
```

**操作步数: 3-5 步**（从首页到完成导入）

#### 场景 2: 快速编辑球员信息

```
分组页面 → Player Card → [⋮] 操作菜单 → 快速编辑
        → Quick Edit Dialog (修改基本信息)
        → 保存 → Toast 通知
```

**操作步数: 3 步**

#### 场景 3: 查看和编辑球员详情

```
分组页面 → Player Card → [⋮] 操作菜单 → 查看详情
        → Detail Dialog (Tabs: 基本信息/比赛记录/历史数据)
        → [编辑] 按钮 → 进入编辑模式
        → 保存 → Toast 通知
```

**操作步数: 3-4 步**

#### 场景 4: 批量导入比赛数据

```
首页 → Shell Bar [数据管理] → 批量导入比赛数据
     → Step 1: 选择文件
     → Step 2: 数据预览
     → Step 3: 字段映射
     → Step 4: 数据验证
     → Step 5: 导入确认
     → 完成 → Toast 通知
```

**操作步数: 3-5 步**

### 7.2 设计决策记录

| 决策 | 选择 | 原因 |
|------|------|------|
| 导入流程 | 5步向导 | 参考 Airtable/Google Contacts，分解复杂任务 |
| 编辑模式 | Dialog + Inline | Dialog 用于完整编辑，Inline 用于快速修改 |
| 删除确认 | Alert Dialog | 防止误操作，显示影响范围 |
| Toast 位置 | 右下角 | 不遮挡操作区域，符合主流习惯 |
| 表格编辑 | 只读 + Dialog | 避免误操作，提供更好的编辑体验 |
